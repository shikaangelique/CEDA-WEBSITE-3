02

The five layers

The site is built in five horizontal layers. A request from a visitor's browser flows down through them and the response flows back up. Each layer has one clear job and depends only on the layer beneath it.

LAYER 1
App Shell
React Router · ThemeProvider · Layout
The outermost wrapper. Decides which page to show based on the URL, holds the persistent header and footer, and provides global context (theme, Supabase client, motion preferences) to everything below it.
↓
LAYER 2
Page Components
9 routes · LandingPage · AboutPage · PublicationsPage · etc.
One file per route. Each page assembles a sequence of section blocks and reusable components into the full view a visitor sees. Pages are composition only. They do not own data fetching or styling logic.
↓
LAYER 3
Reusable Components
TextAppear · Hero · ThematicCard · AssessmentWidget · ContactForm · 12 more
The pieces pages are built from. Each component takes props in and renders markup out. They handle their own animation (text-appear), local state (form fields, filter dropdowns), and visual polish.
↓
LAYER 4
Hooks and Data Layer
usePublications · useAssessment · useContactSubmit · supabaseClient
All conversation with Supabase happens here. Hooks fetch publications, submit contact forms, increment assessment counters. Components never call Supabase directly, they call a hook.
↓
LAYER 5
Backend (Supabase)
4 tables · 3 storage buckets · Row Level Security
The persistent layer. Holds publication metadata, news posts, contact submissions, and assessment counters in Postgres. PDFs and images sit in storage buckets. Public users have read-only access to content tables and write-only access to submission tables.

Hooks and data layer

The only place in the frontend that talks to Supabase. Everything above this layer asks a hook for what it needs and the hook handles the call.

supabaseClient.js
Owns: Single instance of the Supabase client, configured with project URL and anon key.
In
VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY
Out
A configured supabase client instance imported by every hook
The phone line to the database. Configured once, used everywhere.
usePublications.js
Owns: Fetching publications from Supabase, with optional filters and search.
In
filters object (theme, type, year, search)
Out
{ publications, loading, error }
"Give me publications, optionally narrowed down by these filters."
useFeaturedPublications.js
Owns: Fetching the four publications flagged as featured for the landing page.
In
limit (default 4)
Out
{ featured, loading, error }
"Give me the featured publications for the homepage."
usePublicationById.js
Owns: Fetching a single publication and its related publications.
In
id (string)
Out
{ publication, related, loading, error }
"Give me everything about publication X plus its related reports."
useNews.js · useNewsBySlug.js
Owns: Fetching news posts (list or by slug).
In
limit or slug
Out
{ posts, loading, error } or { post, loading, error }
"Give me the news posts" or "give me this specific post."
useAssessment.js
Owns: Submitting visitor answers to the three MCQs and fetching aggregate counters.
In
submit(answers): {q1, q2, q3} option keys
Out
{ submit fn, results: { perOption: {...}, totalResponses } }
Sends the visitor's three picks to Supabase, gets back the percentages to display on the results screen.
useContactSubmit.js
Owns: Posting contact form submissions.
In
{ name, email, phone, organisation, message }
Out
{ submit fn, status: idle | sending | success | error }
Saves the contact form to Supabase. Returns a status flag so the form can show a confirmation.

Database tables

Postgres tables in Supabase. Each table has Row Level Security policies that decide who can read and write.

publications
Owns: Metadata for every publication CEDA has produced.
In
Inserts: admin only (via Supabase dashboard or import script)
Out
Reads: public, no auth required
The list of every report and brief, plus links to the actual PDFs in storage.
news_posts
Owns: News and blog post content.
In
Inserts: admin only
Out
Reads: public, no auth required
The blog posts and news items.
contact_submissions
Owns: Every contact form submission from the public.
In
Inserts: public allowed
Out
Reads: admin only
Inbox for form submissions. The public can write, only the team can read.
assessment_responses
Owns: Aggregate counters for the landing-page MCQ widget. Twelve rows total: 3 questions x 4 options, plus a total-responses row.
In
Increment counters: public allowed via stored procedure
Out
Reads: public, no auth required
The simplest possible setup: counts of how many visitors picked each option, kept fresh by a stored procedure that increments on each submission.

Storage buckets

Files (PDFs, images) live in Supabase Storage, separate from the database. The database holds links to them.

publications/
Owns: All publication PDFs.
In
Uploads: admin only
Out
Public URLs, embedded in publications.file_url
The PDF library.
thumbnails/
Owns: Cover images for every publication.
In
Uploads: admin only
Out
Public URLs, embedded in publications.thumbnail_url
The cover images shown on publication cards.
media/
Owns: Hero images, news cover images, partner logos, team photos.
In
Uploads: admin only
Out
Public URLs, used across the site
Everything visual that is not a publication PDF or thumbnail.

05

The whole thing, in plain English

If you skipped everything above, this is the architecture in five paragraphs.

The CEDA site is a static React app, deployed to Vercel, that loads quickly and then talks to a Supabase backend whenever it needs real content (publications, news, the assessment counters, contact form). When you visit cedaghana.com, Vercel sends you the bundle. The bundle then asks Supabase for whatever the page you're looking at needs.

The frontend is built in five horizontal layers. The shell is the persistent frame. Pages are one file each, one per URL. Reusable components are the building blocks pages compose. Hooks handle every conversation with Supabase. Supabase is the database and storage at the bottom.

The backend is intentionally tiny: four tables and three storage buckets. Publications and news posts are read-public, write-admin. Contact submissions are write-public, read-admin. The assessment widget uses a counter table with twelve rows: three questions times four options, plus a total. Each visitor submission increments four counter rows through one stored procedure call.

The signature visual move is the text-appear animation, a TextAppear wrapper component used everywhere, that splits text into words and animates each one as it scrolls into view. There is no other copy-level animation. Layout-level motion is limited to a sticky header that gains a blur on scroll, plus subtle hover states on cards.

Everything published goes through the same path: an admin uploads a PDF and thumbnail to Supabase storage, then adds a row to the publications table pointing at those files. The frontend picks up the change automatically. No code changes, no redeploy. The team can publish a new report and have it live on the site within minutes.
